# 后端工程能力知识

## 分层架构
考察点：是否理解路由层、业务层、数据层的边界。

回答要点：
- api 层负责 HTTP 请求、响应和异常转换。
- services 层负责业务编排，例如调用 Agent、渲染报告、保存结果。
- schemas 层定义请求响应和 LLM 结构化输出。
- database 层负责 SQLite 读写，避免 SQL 散落在业务代码里。

项目追问：
- `generate_job_match_report` 为什么放在 service 层？
- 如果后续换数据库，哪些代码应该尽量不受影响？

## 同步接口与异步任务
考察点：是否理解长耗时任务对用户体验的影响。

回答要点：
- LLM 分析和联网搜索可能耗时几十秒。
- 同步接口会让前端一直等待，失败时用户体验差。
- 第一版可以用 FastAPI BackgroundTasks，后续再升级 Celery/RQ + Redis。

项目追问：
- BackgroundTasks 的局限是什么？
- 为什么任务状态要保存到数据库？

## 日志和可观测性
考察点：是否理解线上问题排查。

回答要点：
- 日志记录任务失败、模型解析失败、报告保存 ID。
- 数据库保存 parse_status、parse_error、latency_ms 便于后续评估。
- AI 应用应记录原始输出，但要注意简历隐私和脱敏。

项目追问：
- `parse_status` 有哪些状态？分别代表什么？
- 如何统计模型输出结构化成功率？

## API 设计
考察点：是否理解接口可维护性。

回答要点：
- 请求模型和响应模型分开定义。
- 长耗时任务使用创建任务接口和查询任务接口。
- 历史报告接口返回报告详情和岗位样本，实现结果追溯。
